# V1.5 Tech Debt

Do not fix in v1 unless it blocks launch.

## 1. Runtime provider registry

- File: `app/src/lib/deep-link.ts`
- Current state: imports `data/streamers.json` directly.
- Why it is wrong: app runtime should consume `GET /api/library/providers`, not a repo-relative data file.
- Why not fixed in v1: the direct import keeps search-link generation pure and deterministic while Cursor rebuilds the picker.
- V1.5 fix: add a provider-registry service in app state, hydrate it from `/api/library/providers`, and pass the registry into deep-link generation.

- Current state: option A LRU reduces repeated provider calls for hot titles, but a cold page can still fetch provider data for up to 40 titles.
- Why not fixed in v1: no paid infra, no licensed feed, no batch protocol yet.
- V1.5 fix: add a first-class batch provider endpoint or a TMDB-backed background provider prewarm job if usage data proves it is needed.

- Why not fixed in v1: correctness while Cursor wires infinite scroll matters more than edge cache tuning.

## 5. Promote `runtimeMinutes` onto `shared/types.ts` `PersistedTitle`

- Logged by Cursor, 2026-04-23, during picker-ui phase 2.
- Current state: `app/src/types.ts` declares a client-only `PersistedTitleClient` interface extending the frozen `PersistedTitle` with `runtimeMinutes?: number | null`. The field is written at pick-time in the picker (via `/api/title/:tmdbType/:tmdbId`), stored in `UserState.selectedTitles` as a transparent extra JSON property, and read by the slot-edit screen to compute `endTime`.
- Why it is wrong: product code has a runtime-shape field that is absent from the shared contract. Cross-actor code (backend, any future client, any tests importing the shared type) does not see it.
- Why not fixed in v1: `shared/types.ts` is Codex-owned and was frozen during the picker rebuild.
- V1.5 fix: promote `runtimeMinutes?: number | null` onto `PersistedTitle` in `shared/types.ts`. Remove the `PersistedTitleClient` interface. Migrate the store shape (no runtime migration needed — JSON already carries the field).

## 6. `scheduler.ts` `hydrateChannel` ignores `slot.titleId`

- Logged by Cursor, 2026-04-23, during picker-ui phase 2.
- Current state: `hydrateChannel(slots, titles, selectedProviders)` at [app/src/lib/scheduler.ts](/C:/dev/mychannel-universal/app/src/lib/scheduler.ts) L40-41 does `titles[index % titles.length]` — it round-robins the `titles` array into the sorted `slots` array and never consults `slot.titleId`. The week-screen per-slot-edit feature (c5) requires per-slot title binding, so Cursor ships a caller-side `rebuildChannel` adapter in `app/src/lib/channel-adapter.ts` that pre-orders `titles` to match sorted-slot order before calling `hydrateChannel`.
- Why it is wrong: callers outside the week screen who want hydrated channel output must know to sort + pre-order their titles the same way, or they get garbage. The contract leaks.
- Why not fixed in v1: `scheduler.ts` was frozen during the picker rebuild.
- V1.5 fix: promote a `titleId`-aware path into `scheduler.ts`. Simplest shape: `hydrateChannel` looks up each slot's title by `slot.titleId` against the `titles` array, falling back to the current index-based pick when `titleId` is absent (preserves old behaviour). Once landed, delete `channel-adapter.ts` and the inline `clampEndToWindow` copy in `slot-edit.ts`.
